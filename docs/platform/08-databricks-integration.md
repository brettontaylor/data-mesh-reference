# 08 — Databricks & Unity Catalog Integration

The `catalog-adapter` package owns all Databricks/UC interaction behind interfaces,
so the core never couples to a vendor SDK directly.

## 1. Connections & auth

- One or more **workspace connections** (dev/staging/prod), configured in Admin,
  each with: host, OAuth service-principal creds (preferred) or PAT, default SQL
  warehouse, and the UC catalog prefix per environment.
- Least-privilege: the service principal can manage only the catalogs/schemas/jobs
  Harbormaster owns (a dedicated `hbr_*` catalog namespace recommended).
- All calls go through the Databricks SDK with retry/backoff + circuit breaker.

## 2. Unity Catalog two-way sync (D2)

### 2.1 Push (Harbormaster → UC) — model becomes governed UC objects
On registration/deploy, the adapter projects the model into UC:
- **Schemas/tables**: gold tables created/aligned to the PDM (the engine already
  generates DDL; the adapter applies it to UC).
- **Comments**: table/column comments from descriptions.
- **Tags**: classification tier + PII/MNPI + domain + owner as UC tags (so UC-native
  tools and policies can use them).
- **Column masks / row filters**: where UC supports it, generate UC masking
  functions from the access model (belt-and-suspenders with API-layer masking).
- **Lineage hints**: register pipeline/table relationships.

### 2.2 Pull (UC → Harbormaster) — discover & reconcile existing estate
- Enumerate existing UC catalogs/schemas/tables/columns and **import as candidate
  models** (BDM/PDM skeletons) so an org can onboard its current estate, then curate.
- Ingest **UC system tables** (`system.access`, `system.lineage`, `information_schema`)
  to enrich lineage and usage metrics.
- **Reconciliation report**: diff what UC has vs what models declare → surface
  *unmanaged* tables (exist in UC, no model), *undeployed* models (model exists, not
  in UC), and *drift* (UC schema ≠ model). Drift raises governance alerts.

### 2.3 Conflict policy
- **Harbormaster-managed objects** (in the `hbr_*` namespace) are owned by the
  platform; out-of-band UC edits to them are flagged as drift and reverted on next
  deploy (model is SoR).
- **Pre-existing objects** imported from UC are *advisory* until a model adopts them;
  the platform won't mutate unadopted objects.

## 3. Upstream ingestion (product & reference data)

BDMs/PDMs declare a **source** (upstream feed). The platform supports landing
upstream product/ref data into bronze via connectors (adapter pattern):

| Source kind | Mechanism |
|-------------|-----------|
| File drop (CSV/Parquet/JSON) | Autoloader / file-arrival trigger → bronze |
| JDBC / DB | Databricks JDBC ingestion (batch/CDC) → bronze |
| Streaming (Kafka/EventHub) | Structured Streaming → bronze |
| API/SaaS | Scheduled extract task → object store → bronze |
| Existing UC table | Reference an existing UC table directly (no copy) |

The **source registration** (`sources/*.yaml`) carries connection ref (secret),
cadence, schema expectations, and the bronze landing target. Ingestion specifics are
generated by the engine + executed by Databricks; Harbormaster governs and monitors,
it doesn't move bytes itself.

### 3.1 Schema evolution from upstream
- Bronze is schema-on-read/lenient; the **silver** conform step enforces the BDM
  contract. If upstream adds a column, a **schema-drift detector** (compare observed
  bronze schema to BDM) opens a *suggested ChangeSet* ("upstream added `x`; add to
  BDM?") rather than silently breaking — governed evolution.
- Incompatible upstream changes (type narrowing, dropped key) quarantine the batch
  and alert the owner.

## 4. Reconciliation engine (the safety net)

A scheduled job reconciles three planes and reports divergence:
1. **Git ↔ Postgres projection** (definitions vs index) — rebuild if drifted.
2. **Models ↔ UC** (declared vs deployed) — drift alerts + redeploy.
3. **Static lineage ↔ observed (OpenLineage)** — confirm/enrich, flag surprises.

Each reconciliation pass is idempotent and audited; an operator dashboard shows the
last pass, divergences, and one-click remediation (redeploy / rebuild / open change).

## 5. Cost & performance controls

- Compute is config per environment (job clusters, photon, autoscale bounds, pools);
  no hardcoded cluster sizes.
- Per-connector/workspace concurrency caps prevent runaway parallel deploys/runs.
- DBU/cost estimates captured per run and shown on dashboards; budget alerts
  configurable per domain/product.

## 6. Edge cases (Databricks-specific)

| Case | Handling |
|------|----------|
| Workspace unreachable / token expired | Degrade orchestration only; queue + retry; alert; auto-rotate token |
| UC permission denied on managed object | Clear diagnostic; surface required grant; never silently skip |
| Partial bundle deploy failure | Roll back bundle; model state untouched (Git intact); show diff |
| Out-of-band UC schema edit | Drift alert; revert on next deploy (managed) or advise (unmanaged) |
| Duplicate/renamed table collision | Name+content hash diffing; block ambiguous renames pending review |
| Large catalog import (10k+ tables) | Paged, throttled import; background job; progressive projection |
| Cross-region workspace | Residency-aware deploy targets; per-region projection shard |
| DLT vs Workflows feature gaps | Capability flags per workspace; generator falls back gracefully |

## 7. Databricks App packaging (optional, D3)

The same image is bundled as a Databricks App (`deploy/databricks-app/`) so the
control plane can run *inside* the workspace perimeter, inheriting Databricks auth
and networking, with Postgres as a managed instance. Identical features; different
host. Chosen when the org wants everything within Databricks' boundary.
