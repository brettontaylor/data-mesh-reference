# 05 — APIs, SDKs, CLI & Open Standards

Surfaces for **producing to** and **consuming from** the stack. Everything is
schema-first: the API is described by OpenAPI generated from the same JSON Schema
the engine publishes, so SDKs and docs never drift.

## 1. API design

- **Style:** REST (resource-oriented) as the primary contract; an optional GraphQL
  read layer for graph-shaped queries (lineage/impact) where REST is awkward.
- **Versioned:** `/api/v1/...`; additive changes only within a major; deprecations
  announced via `Sunset` headers and the changelog.
- **AuthN:** OIDC bearer tokens (humans, SSO) and **API keys / service principals**
  (machines), scoped to roles + domains. All keyed access is rate-limited and
  audited.
- **AuthZ:** every response is filtered by the caller's clearance — attribute-level
  masking (PII/MNPI + tier) applied server-side, identical to the demo engine.
- **Pagination:** cursor-based; **filtering** via documented query params;
  **field selection** to minimize payloads.
- **Errors:** RFC 9457 `application/problem+json` with stable `type` URNs.
- **Idempotency:** mutating endpoints accept an `Idempotency-Key`.

### 1.1 Resource map (v1)

```
# Discovery / consumption (read)
GET  /api/v1/domains
GET  /api/v1/models?kind=&domain=&status=&q=
GET  /api/v1/models/{kind}/{id}                 # incl. version, schema, lineage refs
GET  /api/v1/models/{kind}/{id}/versions
GET  /api/v1/models/{kind}/{id}/lineage?direction=upstream|downstream&depth=
GET  /api/v1/products                            # data products
GET  /api/v1/products/{id}                        # contract, SLA, endpoints, health
GET  /api/v1/products/{id}/data?role=…&filter=…   # access-governed sample/serving
GET  /api/v1/semantic/{model}/query              # measure/dimension query (governed)
GET  /api/v1/glossary, /api/v1/glossary/{term}
GET  /api/v1/registry                            # full model registry + versions
GET  /api/v1/lineage/search?urn=                 # column-level lineage graph
GET  /api/v1/search?q=                           # cross-entity search

# Production / governance (write — RBAC + SoD enforced)
POST /api/v1/changesets                          # propose (opens branch+PR)
GET  /api/v1/changesets/{id}                      # diff, impact, gates, approvals
POST /api/v1/changesets/{id}/submit              # draft → in_review
POST /api/v1/changesets/{id}/approve|reject       # checker (SoD enforced)
POST /api/v1/changesets/{id}/merge                # approved → active
POST /api/v1/models/{kind}/{id}/validate          # dry-run validation
POST /api/v1/models/{kind}/{id}/simulate          # impact + generated-artifact preview

# Orchestration (platform_eng)
POST /api/v1/pipelines/{id}/deploy?env=
POST /api/v1/pipelines/{id}/trigger
GET  /api/v1/pipelines/{id}/runs, /runs/{runId}

# Subscriptions / events
POST /api/v1/subscriptions                        # subscribe app/consumer to a product
GET  /api/v1/events?since=                         # event feed (audit-safe subset)
POST /api/v1/webhooks                              # register outbound webhook

# Platform / admin
GET  /api/v1/health, /api/v1/ready, /api/v1/version
GET  /api/v1/audit?actor=&subject=&since=          # governance/admin only
GET  /openapi.json                                 # machine-readable spec
```

### 1.2 Contracts as first-class published artifacts

For each model/product the platform serves machine-readable contracts (the
"publish" half), generated by the engine:

- `…/schema.json` — JSON Schema of the entity/product
- `…/openapi.json` — the consumption endpoints for that product
- `…/context.jsonld` — JSON-LD semantic context (for semantic-web/catalog federation)
- `…/lineage.json` — OpenLineage-formatted lineage
- `/.well-known/harbormaster.json` — capability descriptor + standard version + changelog

## 2. SDKs

### 2.1 TypeScript SDK (`packages/sdk-ts`)
Typed client generated from OpenAPI + hand-written ergonomic helpers:

```ts
import { Harbormaster } from "@harbormaster/sdk";
const hbr = new Harbormaster({ baseUrl, token });           // OIDC or API key

// consume
const product = await hbr.products.get("trading_activity");
const rows = await hbr.products.data("trading_activity", { role: "analyst" });
const lineage = await hbr.lineage.upstream("hbr:field:trading.trade.notional");

// produce (governed)
const cs = await hbr.changesets.propose({
  title: "Add settlement_date to trade",
  edits: [{ kind: "bdm", id: "trade", patch: { addField: {/*…*/} } }],
});
await hbr.changesets.submit(cs.id);
```

### 2.2 Python SDK (`sdk-python`)
Thin REST client + pydantic models (for data scientists / Databricks notebooks):

```python
from harbormaster import Client
hbr = Client(base_url, token)
df = hbr.product("trading_activity").to_pandas(role="risk")   # access-governed
schema = hbr.model("bdm", "trade").schema
```

Both SDKs respect classification masking transparently and surface clear errors when
a field/measure is unavailable to the caller's clearance.

## 3. CLI (`hbr`)

For modelers, platform engineers, and CI. Wraps the engine locally + the API
remotely.

```
hbr init                      # scaffold a models repo / connect to an existing one
hbr validate [path]           # engine validation locally (pre-commit)
hbr diff <model>              # control-surface diff vs registered baseline
hbr propose -m "…"            # branch + commit + open PR + create ChangeSet
hbr models | model <id>       # registry list / detail (existing dmref commands)
hbr register                  # update the lock (post-merge / release)
hbr simulate <model>          # impact + generated-artifact preview
hbr generate                  # write generated/ artifacts locally
hbr pipeline deploy|trigger|runs --env staging
hbr lineage <urn> [--down]    # lineage/impact from the CLI
hbr login                     # OIDC device flow → token cache
```

`dmref` remains a documented alias of `hbr` for the engine subset (no breakage).

## 4. Events & webhooks (push)

- **Transactional outbox** → Postgres `NOTIFY` (in-process subscribers) and an
  outbound dispatcher for **webhooks** and optional **Kafka** topics.
- Event types: `model.registered`, `version.bumped`, `changeset.approved/merged`,
  `pipeline.run.completed`, `product.published`, `classification.changed`,
  `sla.breached`, `model.deprecated`.
- Consumers subscribe to products and receive breaking-change notices; apps can
  react (e.g., re-pull a schema) automatically.
- Delivery: at-least-once with signed payloads (HMAC), retries with backoff, and a
  dead-letter view for ops.

## 5. Open standards (D6) — federation, not lock-in

| Standard | Used for |
|----------|---------|
| **OpenAPI 3.1** | API description; SDK + docs generation |
| **JSON Schema** | Model/entity schemas; validation; publication |
| **JSON-LD** | Semantic context for catalog/semantic-web federation |
| **OpenLineage** | Lineage in/out — interoperate with any lineage backend |
| **SQL/DDL** | Generated warehouse artifacts (Snowflake/Databricks SQL) |
| **dbt/DLT** | Generated transformation assets |
| **SCIM / OIDC / SAML** | Identity federation |
| **CloudEvents** | Event envelope for webhooks/Kafka |

Because the platform speaks these, it can **complement or be complemented by**
Collibra/Alation/Atlan/Purview later (D6 "net-new, integrate via open standards")
without re-architecture.

## 6. Backwards/forwards compatibility policy

- API additive within a major; removals only at a major with a deprecation window
  and `Sunset` headers.
- Generated contracts are versioned with the model; consumers pin a product version
  and are notified before a breaking bump (impact + events).
- SDKs follow semver aligned to the API major.
